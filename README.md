# Игра "Разгадай слово"

Игра, в которой пользователь должен угадать исходное слово из перемешанных букв. За каждый правильный ответ начисляются очки (20).

---

## Технологии

- Kotlin
- Jetpack Compose
- ViewModel
- StateFlow
- JUnit

---

## Функциональность приложения

- Отображение перемешанного слова
- Поле ввода для ответя пользователя
- Проверка правильности введенного слова
- Подсчет очков (20 очков за правильный ответ)
- Кнопка `Пропустить` для перехода к следующему слову без начисления очков
- Кнопка `Играть снова` для перезапуска

## Контрольные вопросы

1. Что такое ViewModel и зачем он нужен?
    - **ViewModel** - это специальный класс из Android Jetpack, который хранит данные вашего UI отдельно от Composable-функций.
    - Без ViewModel при повороте экрана Android стирает Activity и создает заново - все данные теряются и обнуляются.
    - При повороте все переменные сбрасываются к начальным значениям

2. В чём разница между MutableStateFlow и StateFlow?
    - **MutableStateFlow** - изменяемый поток
    - **StateFlow** - неизменяемый поок
    - **_uiState и uiState** - это паттерн "backing property" - защита от случайного изменения состояния UI. ViewModel меняет `_uiState`, а UI читает только `uiState`.
    - **Backing property** - Приватное изменяемо поле используется внутри класса, а наружу отдается его неизменяемая копия.

3.  Почему в ViewModel нельзя хранить Context или View?
   - ViewModel живет дольше, чем Activity/Composable. Если сохранить ссылку на Activity, это вызовет утечеку памяти - Activity не удалится сборщиком мусора, из-за этого может привести к переполнению памяти и крашу приложения.

4. Чем StateFlow отличается от remember и rememberSaveable?
   - **remember** - сохраняет состояни только во время текущего существования Composable, теряется при повороте экрана
   - **rememberSaveable** - сохраняет состояние при повороте, но теряется при закрытии приложения
   - **StateFlow** - сохраняется при повороте и живет дольше самой Activity
   - Когда что использовать:
     - `remember` - для временного UI-состояния
     - `rememberSaveable` - для небольших данных, который нужно сохранить при повороте
     - `StateFlow` в ViewModel - для безнес-логики и данных, которые должны жить вне UI

5.  Что такое data class в Kotlin и зачем он используется для UI State?
   - **Data class** - спец. класс, предназначенный для хранения данных. Он автоматически генерирует методы: `equals()`, `hashCode()`, `toString()`, `copy()`.
   - **Функция copy()** - позвоялет создать копию объекта с изменением только нужных полей. 

6. Как работает collectAsState() в Compose?
    - `collectAsState()` - это функция, которая подписывается на StateFlow и превращает его в State, понятный Compose.
    - При изменении значения в StateFlow, `collectAsState()` получает уведомление и вызывает перерисовку всех Composable - функций, которые используют это состояние.

7. Зачем разделять код на пакеты data и ui?
   - `data` - модели данных и бизнес-логика
   - `ui` - компоненты интерфейса
    - Какие принципы архитектуры это поддерживает:
      - Разделение ответственности - каждый пакет отвечает за свое
      - Упрощение тестировки - можно тестировать ViewModel без UI

8. Что происходит с ViewModel, когда приложение полностью закрывается (process death)?
    - При закрытии приложения ViewModel уничтожается и данные теряются.
    - Данные в ViewModel не сохраняются - они хранятся только в оперативной памяти.
    - Какие альтернативы есть для долговременного хранения:
        - `SharedPreferences` - для простых настроек и небольших данных
        - `File storage` - для файлов
        - `DataStore` - современная замена SharedPreferences
